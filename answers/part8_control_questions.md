# Часть 8: Контрольные вопросы

1. Сформулируйте принцип IoC своими словами. Чем он отличается от обычного процедурного подхода?

Ответ: IoC (Inversion of Control) — принцип, при котором объект не создаёт свои зависимости сам, а получает их извне. В обычном процедурном подходе класс сам контролирует создание нужных ему объектов: `new ServiceImpl()`. При IoC управление передаётся контейнеру (Spring): он сам создаёт объекты, управляет их жизненным циклом и внедряет зависимости. Код становится слабосвязанным — класс зависит от абстракции (интерфейса), а не от конкретной реализации.

2. Назовите три способа DI в Spring. Какой из них предпочтителен и почему?

Ответ: Три способа DI в Spring:
- **Constructor injection** — зависимость передаётся через конструктор (`GreetingController`, `StudentServiceImpl`);
- **Setter injection** — зависимость устанавливается через `@Autowired`-сеттер (`SetterInjectionDemo`);
- **Field injection** — `@Autowired` ставится прямо на поле (`FieldInjectionDemo`).

Предпочтителен **constructor injection**: зависимости могут быть `final`, объект всегда создаётся в полном состоянии, легко тестировать без Spring-контекста (достаточно `new`), и Spring не требует явного `@Autowired` при одном конструкторе.

3. Чем отличаются `@Component`, `@Service`, `@Repository`, `@Controller`?

Ответ: Все четыре аннотации являются специализациями `@Component` и регистрируют класс как бин в Spring-контексте. Разница — семантическая и функциональная:
- `@Component` — универсальная аннотация для любого бина без выраженной роли;
- `@Service` — маркирует бин сервисного слоя с бизнес-логикой (используется в `StudentServiceImpl`);
- `@Repository` — маркирует DAO/репозиторий, дополнительно включает трансляцию JPA-исключений в Spring DataAccessException;
- `@Controller` — маркирует MVC-контроллер, методы которого возвращают имена view-шаблонов (`StudentWebController`).

4. Чем `@RestController` отличается от `@Controller`?

Ответ: `@RestController` — это `@Controller` + `@ResponseBody`. Это означает, что возвращаемые значения всех методов автоматически сериализуются в JSON (или XML) через HttpMessageConverter и записываются прямо в тело HTTP-ответа, минуя ViewResolver. `@Controller` без `@ResponseBody` интерпретирует возвращаемую строку как имя шаблона для рендеринга (Thymeleaf). В проекте `StudentRestController` помечен `@RestController`, а `StudentWebController` — `@Controller`.

5. Что делает аннотация `@SpringBootApplication`?

Ответ: `@SpringBootApplication` — составная аннотация, объединяющая три:
- `@SpringBootConfiguration` (специализация `@Configuration`) — помечает класс как источник конфигурации бинов;
- `@EnableAutoConfiguration` — включает механизм автоконфигурации Spring Boot, который по classpath автоматически настраивает компоненты (Tomcat, JPA, Security и др.);
- `@ComponentScan` — запускает сканирование пакета текущего класса и всех подпакетов для обнаружения `@Component`, `@Service`, `@Repository`, `@Controller`.

6. Что такое starter-модуль? Приведите 3 примера и опишите, что они подключают.

Ответ: Starter — это Maven/Gradle-зависимость, которая агрегирует набор библиотек и автоконфигураций для конкретной функциональности. Не нужно вручную перечислять транзитивные зависимости — достаточно добавить один starter. Примеры из `pom.xml` проекта:
- `spring-boot-starter-web` — подключает Spring MVC, встроенный Tomcat, Jackson для JSON;
- `spring-boot-starter-data-jpa` — подключает Hibernate, Spring Data JPA, управление транзакциями;
- `spring-boot-starter-security` — подключает Spring Security с базовой HTTP-аутентификацией и фильтром безопасности.

7. Что такое `ApplicationContext` и чем он отличается от `BeanFactory`?

Ответ: `BeanFactory` — базовый IoC-контейнер Spring: создаёт и хранит бины, внедряет зависимости, поддерживает ленивую инициализацию. `ApplicationContext` расширяет `BeanFactory` и добавляет: поддержку i18n (`MessageSource`), публикацию и подписку на события (`ApplicationEvent`), интеграцию с AOP, загрузку ресурсов, eager-инициализацию singleton-бинов при старте. В Spring Boot всегда используется `ApplicationContext` (`AnnotationConfigServletWebServerApplicationContext` для веб-приложений).

8. Какие scope бинов вы знаете и в чём их различие?

Ответ:
- **singleton** (по умолчанию) — один экземпляр бина на весь ApplicationContext; все зависимости получают одну и ту же ссылку. Используется в `StudentServiceImpl`, `StudentRestController` и др.;
- **prototype** — новый экземпляр создаётся каждый раз при запросе бина из контейнера;
- **request** — один экземпляр на HTTP-запрос (только в веб-контексте);
- **session** — один экземпляр на HTTP-сессию пользователя (только в веб-контексте);
- **application** — один экземпляр на `ServletContext` (весь жизненный цикл приложения).

9. Объясните AOP: Aspect, JoinPoint, Pointcut, Advice.

Ответ: AOP (Aspect-Oriented Programming) позволяет вынести сквозную логику (логирование, транзакции, безопасность) в отдельный модуль — аспект. Термины на примере `LoggingAspect`:
- **Aspect** — класс с логикой аспекта, помеченный `@Aspect @Component` (`LoggingAspect`);
- **JoinPoint** — конкретная точка выполнения программы, в которую может быть встроен совет; в Spring — вызов метода бина;
- **Pointcut** — выражение, определяющее набор JoinPoint-ов, к которым применяется совет: `execution(* lecture.seven.student.service.*.*(..))`;
- **Advice** — действие, выполняемое в точке среза: `@Before` (до метода), `@AfterReturning` (после успешного возврата), `@Around` (вокруг вызова).

10. Что делает `@PathVariable` и чем отличается от `@RequestParam`?

Ответ: `@PathVariable` извлекает значение из сегмента URL-пути. Например, в `@GetMapping("/{id}")` аннотация `@PathVariable Long id` получит `1` из запроса `GET /api/students/1`. `@RequestParam` извлекает значение из query string (параметры после `?`). Например, `@RequestParam String q` получит `Sash` из `GET /api/students/search?q=Sash`. `@PathVariable` используется для идентификации ресурса, `@RequestParam` — для фильтрации, сортировки, пагинации.

11. Объясните цепочку `Controller → Service → Repository`.

Ответ: Это трёхслойная архитектура (Layered Architecture):
- **Controller** (`StudentRestController`) — принимает HTTP-запрос, валидирует входные данные, делегирует бизнес-логику сервису, формирует HTTP-ответ. Не содержит бизнес-логики и не работает с БД напрямую;
- **Service** (`StudentServiceImpl`) — содержит бизнес-логику, управляет транзакциями (`@Transactional`), оркестрирует вызовы репозиториев;
- **Repository** (`StudentRepository`) — отвечает только за доступ к данным: CRUD-операции, запросы к БД через JPA. Не содержит бизнес-логики.

12. Что такое `JpaRepository`? Как Spring Data генерирует методы вида `findByNameContainingIgnoreCase`?

Ответ: `JpaRepository<Student, Long>` — интерфейс Spring Data JPA, предоставляющий готовые CRUD-методы (`findAll`, `save`, `deleteById`, `count` и др.) без написания реализации. При запуске приложения Spring Data анализирует имена объявленных методов по соглашению об именовании: разбивает на ключевые слова (`findBy` → SELECT WHERE, `Containing` → LIKE '%value%', `IgnoreCase` → UPPER()), транслирует в JPQL-запрос и генерирует реализацию через динамический прокси. Разработчику не нужно писать ни SQL, ни реализацию вручную.

13. Как Thymeleaf получает данные от контроллера и подставляет их в шаблон?

Ответ: Контроллер добавляет данные в объект `Model` методом `model.addAttribute("students", service.findAll())`. Thymeleaf получает доступ к этим данным через переменные контекста по имени атрибута. В шаблоне `students.html` `th:each="s : ${students}"` итерирует список, `th:text="${s.name}"` подставляет значение поля, `th:object="${student}"` привязывает форму к объекту, а `th:field="*{name}"` генерирует атрибуты `input`-поля. Всё это происходит на сервере при рендеринге — браузер получает уже готовый HTML.

14. Что такое `SecurityFilterChain`?

Ответ: `SecurityFilterChain` — это цепочка Servlet-фильтров Spring Security, через которую проходит каждый HTTP-запрос до попадания в `DispatcherServlet`. Фильтры последовательно проверяют аутентификацию (Basic Auth, форм-логин, JWT), авторизацию (соответствие URL правилам из `.authorizeHttpRequests`), CSRF-защиту и другие аспекты безопасности. В `SecurityConfig` метод `filterChain(HttpSecurity http)` конфигурирует и возвращает этот бин, определяя правила доступа для каждого URL-паттерна.

15. Зачем нужен `PasswordEncoder` и почему пароли нельзя хранить открыто?

Ответ: Хранение паролей в открытом виде катастрофически опасно: при утечке БД злоумышленник немедленно получит все пароли пользователей, которые часто используются повторно на других сервисах. `PasswordEncoder` (в проекте — `BCryptPasswordEncoder`) применяет однонаправленное хэширование с солью: из хэша невозможно восстановить исходный пароль, а соль делает невозможными атаки по радужным таблицам. При проверке пароля Spring Security хэширует введённое значение и сравнивает с хранимым хэшем.

16. Опишите алгоритм JWT-аутентификации.

Ответ:
1. Клиент отправляет логин и пароль на `POST /auth/login`;
2. Сервер проверяет учётные данные, создаёт JWT-токен (header.payload.signature) и подписывает его секретным ключом;
3. Клиент сохраняет токен (localStorage или cookie) и отправляет его в каждом запросе в заголовке `Authorization: Bearer <token>`;
4. Сервер извлекает токен из заголовка, проверяет подпись и срок действия (`exp`), извлекает `subject` (userId/username) из payload;
5. Spring Security помещает пользователя в `SecurityContext` — запрос авторизован без обращения к БД.

17. Преимущество JWT перед session-based authentication для REST API?

Ответ: Session-based аутентификация требует хранения сессий на сервере (в памяти или Redis), что усложняет горизонтальное масштабирование — при нескольких инстансах нужно липкие сессии или централизованное хранилище. JWT — stateless: сервер не хранит состояния, любой инстанс может проверить токен по подписи. Это упрощает масштабирование, подходит для микросервисов и мобильных клиентов. Недостаток JWT — сложность инвалидации токена до истечения `exp`.

18. Зачем нужна `@Valid`?

Ответ: `@Valid` на параметре метода контроллера активирует Bean Validation для переданного объекта. Без неё аннотации `@NotBlank`, `@Size` и другие на полях `StudentRequest` будут полностью проигнорированы — данные попадут в сервис и БД без какой-либо проверки. При нарушении ограничений Spring бросает `MethodArgumentNotValidException`, которое `GlobalExceptionHandler` перехватывает и возвращает клиенту `400 Bad Request` с детальным описанием ошибок по каждому полю.

19. Перечислите 5 проблем возврата `@Entity` напрямую из REST.

Ответ:
- **Утечка внутренней структуры БД** — клиент видит все поля entity, включая технические (версии, внутренние ID), что нарушает инкапсуляцию;
- **LazyInitializationException** — при попытке Jackson сериализовать lazy-коллекции после закрытия JPA-сессии;
- **Бесконечная рекурсия** — двунаправленные `@OneToMany`/`@ManyToOne` связи вызывают бесконечный цикл сериализации и `StackOverflowError`;
- **Отсутствие версионирования API** — изменение схемы БД сразу ломает API-контракт клиентов;
- **Невозможность гибкой валидации** — у entity нет разделения между правилами при создании и обновлении; нельзя запретить клиенту передать `id` при POST-запросе.

20. `@RestControllerAdvice` vs `@ControllerAdvice`?

Ответ: `@ControllerAdvice` применяет глобальные обработчики исключений ко всем контроллерам, но возвращаемые объекты интерпретируются как имена view-шаблонов (как в обычном `@Controller`). `@RestControllerAdvice` добавляет `@ResponseBody` — возвращаемые объекты автоматически сериализуются в JSON. В проекте `GlobalExceptionHandler` использует `@RestControllerAdvice`, поэтому методы возвращают объекты `ErrorResponse`, которые Jackson преобразует в JSON-ответ клиенту.

21. Почему `@Transactional` лучше ставить на сервисном слое?

Ответ: Сервисный метод представляет собой одну атомарную бизнес-операцию, которая может включать несколько вызовов репозитория. Если поставить `@Transactional` на уровне репозитория, каждый вызов будет отдельной транзакцией — потеряется атомарность. Например, в `saveTwoOneBroken` сохраняются два студента: если второй невалиден, первый должен откатиться. Это работает только при одной транзакции на весь метод сервиса. Контроллер слишком высокий уровень — он не должен знать о транзакционной семантике.

22. Что такое self-invocation в `@Transactional`?

Ответ: Self-invocation — это вызов `@Transactional`-метода из другого метода того же класса через `this`. В `TxPitfallService` метод `outerWithoutAnnotation` вызывает `this.innerWithTransaction(...)`. Spring управляет транзакциями через AOP-прокси: при вызове извне прокси перехватывает вызов и оборачивает его в транзакцию. Но `this.innerWithTransaction()` обходит прокси и вызывает метод напрямую — транзакция не открывается. Результат: даже при `@Transactional` на `innerWithTransaction` откат не произойдёт, оба студента сохранятся раздельно.

23. Откатится ли транзакция по умолчанию при checked-исключении, например `IOException`?

Ответ: Нет. По умолчанию Spring откатывает транзакцию только при `RuntimeException` (unchecked) и `Error`. Checked-исключения (наследники `Exception`, не являющиеся `RuntimeException`) не вызывают автоматического отката — транзакция будет зафиксирована. Чтобы откатить при `IOException`, нужно явно указать: `@Transactional(rollbackFor = IOException.class)`. В проекте `saveTwoOneBroken` бросает `IllegalStateException` (unchecked) — откат происходит корректно.
