# Часть 2: IoC и DI

## Задание 2.1

1. Как Spring знает, какой бин подставить в `GreetingController`?

Ответ: Spring сканирует classpath на наличие классов, аннотированных `@Component`, `@Service`, `@Repository`, `@Controller` и их производными. `EnglishGreetingService` помечен `@Service`, поэтому Spring регистрирует его как бин в IoC-контейнере. В `GreetingController` объявлен конструктор, принимающий `GreetingService` — Spring видит этот тип, находит подходящий бин (`EnglishGreetingService` реализует `GreetingService`) и автоматически передаёт его при создании контроллера (constructor injection без явного `@Autowired`, так как конструктор единственный).

2. Что произойдёт, если убрать `@Service`?

Ответ: Spring не зарегистрирует `EnglishGreetingService` как бин в контексте приложения. При запуске IoC-контейнер не найдёт ни одного бина, реализующего интерфейс `GreetingService`, и выбросит исключение `NoSuchBeanDefinitionException` — приложение не запустится.

3. Что произойдёт при двух реализациях `GreetingService` без дополнительной настройки?

Ответ: Spring обнаружит два бина одного типа (`GreetingService`) и не сможет однозначно определить, какой из них внедрить в `GreetingController`. Это приведёт к исключению `NoUniqueBeanDefinitionException` при старте. Решить конфликт можно тремя способами: пометить одну из реализаций аннотацией `@Primary` (как это сделано в `EnglishGreetingService`), использовать `@Qualifier("beanName")` на точке внедрения, либо настроить выбор программно через `@Configuration`.

## Задание 2.2

1. Какой способ DI предпочтительнее и почему?

Ответ: Предпочтительнее **constructor injection** (внедрение через конструктор). Он гарантирует, что объект создаётся уже со всеми обязательными зависимостями — невозможно получить бин в неполном состоянии. Кроме того, зависимости могут быть объявлены `final`, что обеспечивает иммутабельность и потокобезопасность. Такие классы легко тестировать без Spring-контекста: достаточно создать объект через `new`, передав нужные зависимости напрямую.

2. Почему поле при constructor injection может быть `final`?

Ответ: В Java поле `final` должно быть инициализировано ровно один раз — либо при объявлении, либо в конструкторе. Constructor injection присваивает зависимость именно в конструкторе (`this.service = service`), поэтому требование компилятора выполняется. При field injection (`@Autowired` на поле) Spring инжектирует значение уже после вызова конструктора через рефлексию, обходя стандартный механизм Java — объявить такое поле `final` невозможно, так как компилятор не допустит присвоения значения вне конструктора.

3. Какой способ труднее всего тестировать без Spring-контекста?

Ответ: Труднее всего тестировать **field injection** (`@Autowired` прямо на поле, как в `FieldInjectionDemo`). Поскольку поле приватное и нет ни конструктора, ни сеттера для его установки, в unit-тесте без Spring-контейнера передать зависимость стандартными средствами Java невозможно. Приходится прибегать к рефлексии вручную или подключать дополнительные библиотеки (например, `ReflectionTestUtils` из Spring Test), что усложняет и замедляет написание тестов.
