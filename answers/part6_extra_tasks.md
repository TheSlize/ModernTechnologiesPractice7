# Часть 6: Дополнительные задания

## 6.1 Custom JpaRepository methods

Ответ / результат: В `StudentRepository` добавлены три кастомных метода Spring Data JPA:
- `findByNameContainingIgnoreCase(String namePart)` — ищет студентов, у которых поле `name` содержит переданную подстроку (без учёта регистра);
- `findBySurnameContainingIgnoreCase(String surnamePart)` — аналогично по полю `surname`;
- `countByName(String name)` — возвращает количество студентов с точным совпадением имени.

Spring Data JPA автоматически генерирует реализацию этих методов во время запуска приложения, разбирая имя метода по ключевым словам (`findBy`, `Containing`, `IgnoreCase`, `count`) и транслируя их в соответствующие JPQL-запросы. Писать SQL или реализацию вручную не требуется.

## 6.2 AOP logging

Какие записи появились в логе?

Ответ: `LoggingAspect` перехватывает все методы из пакета `lecture.seven.student.service.*`. При вызове, например, `GET /api/students` в консоли появятся две записи:

INFO LoggingAspect : >>> StudentService.findAll()
INFO LoggingAspect : <<< findAll => [Student(id=1, name=Sash, ...), ...]


Аннотация `@Before` срабатывает до выполнения метода сервиса и логирует его сигнатуру (`>>> StudentService.findAll()`). Аннотация `@AfterReturning` срабатывает после успешного возврата и логирует имя метода и возвращаемое значение (`<<< findAll => ...`).

## 6.3 AppInitializer

Что будет без `@PostConstruct`? Что будет без `@Component`?

Ответ:
- **Без `@PostConstruct`**: метод `init()` не будет вызван автоматически после создания бина. База данных останется пустой при первом запуске — начальные данные (Sash Brown, Jone Smith, Deema Johnson, Ekatrena Wilson) не будут добавлены.
- **Без `@Component`**: Spring не зарегистрирует `AppInitializer` как бин в контексте приложения. Класс просто не будет создан и управляться IoC-контейнером — `@PostConstruct` не сработает ни при каких условиях, инициализация БД не произойдёт совсем.

## 6.4 Test

1. Что делает `@WebMvcTest`?

Ответ: `@WebMvcTest(StudentRestController.class)` поднимает только веб-слой Spring MVC — без полного контекста приложения, без JPA, без базы данных. Создаётся только указанный контроллер, фильтры, `DispatcherServlet` и сопутствующая инфраструктура MVC. Это делает тест быстрым и изолированным: проверяется только логика HTTP-слоя (маппинги, статус-коды, сериализация JSON), а зависимости сервисного слоя подменяются моками.

2. Зачем нужен `@MockitoBean`?

Ответ: `@MockitoBean` создаёт Mockito-мок для `StudentService` и регистрирует его как бин в Spring-контексте теста вместо реальной реализации. Поскольку `@WebMvcTest` не поднимает сервисный и репозиторный слои, Spring не нашёл бы бин `StudentService` для инжекции в контроллер и упал бы с ошибкой. `@MockitoBean` решает эту проблему и одновременно позволяет задавать поведение через `when(...).thenReturn(...)` для каждого теста отдельно.

3. Что делает `@WithMockUser`?

Ответ: `@WithMockUser` добавляет в `SecurityContext` фиктивного аутентифицированного пользователя с заданными ролями на время выполнения теста — без реального логина. Это позволяет тестировать эндпоинты, защищённые Spring Security (`@PreAuthorize`, `hasRole`), не поднимая реальную аутентификацию. В данном проекте тест использует `@AutoConfigureMockMvc(addFilters = false)`, что полностью отключает фильтры Security, поэтому `@WithMockUser` здесь не применяется — но при включённых фильтрах он был бы необходим для тестов с проверкой авторизации.
