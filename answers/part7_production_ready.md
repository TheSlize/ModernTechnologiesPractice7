# Часть 7: Production-ready практики

## 7.1 Validation

1. Что произойдёт, если убрать `@Valid`?

Ответ: Если убрать `@Valid` с параметра `@RequestBody StudentRequest request` в контроллере, Spring не будет запускать Bean Validation для входящего объекта. Аннотации `@NotBlank` и `@Size` на полях `StudentRequest` будут проигнорированы — запрос с пустым именем (`""`) или фамилией из одного символа (`"A"`) успешно пройдёт в сервис и сохранится в БД. `MethodArgumentNotValidException` не будет брошено, и `GlobalExceptionHandler` не отработает.

2. Чем `@NotNull` отличается от `@NotBlank` для `String`?

Ответ: `@NotNull` проверяет только то, что значение не равно `null` — строка `""` или `"   "` (пробелы) пройдут валидацию. `@NotBlank` проверяет сразу три условия: значение не `null`, не пустая строка и содержит хотя бы один непробельный символ. В `StudentRequest` используется `@NotBlank` — это правильно для полей `name` и `surname`, так как имя из одних пробелов недопустимо.

3. Когда запускается валидация?

Ответ: Валидация запускается в момент привязки и десериализации тела запроса — сразу после того, как Jackson преобразовал JSON в объект `StudentRequest`, но до того, как метод контроллера получит управление. Если хотя бы одно ограничение нарушено, Spring бросает `MethodArgumentNotValidException`, которое перехватывает `GlobalExceptionHandler` и возвращает клиенту `400 Bad Request` с описанием ошибок по полям.

## 7.2 DTO and Mapper

1. Зачем разделять Entity и DTO?

Ответ: Entity (`Student`) — это JPA-объект, напрямую отображённый на таблицу БД. Возврат его из REST API опасен: можно случайно раскрыть внутренние поля (пароли, технические метаданные), нарушить инкапсуляцию или получить ошибку сериализации lazy-связей. DTO (`StudentRequest`, `StudentResponse`) — это контракт API, независимый от схемы БД. Разделение позволяет менять структуру БД без изменения API и наоборот, а также применять разные правила валидации на входе и формат ответа на выходе.

2. Что может случиться при возврате lazy entity напрямую?

Ответ: Если у `Student` есть lazy-коллекции (например, `@OneToMany`) и entity возвращается напрямую из контроллера, Jackson попытается сериализовать эти коллекции уже после закрытия сессии Hibernate. Это вызовет `LazyInitializationException` — «no Session». Даже если сессия ещё открыта, Jackson может рекурсивно обходить двунаправленные связи и уйти в бесконечный цикл, что приведёт к `StackOverflowError` или `JsonMappingException`.

3. Преимущества `record` для DTO?

Ответ: Java `record` (использован в `StudentRequest` и `StudentResponse`) автоматически генерирует конструктор со всеми полями, геттеры, `equals()`, `hashCode()` и `toString()`. Поля record неизменяемы (`final`) по умолчанию — это гарантирует, что DTO не будет случайно модифицирован после создания. Код становится лаконичнее: вместо десятков строк с геттерами/сеттерами — одна строка объявления. Также record явно сигнализирует читателю кода, что это «чистый носитель данных» без бизнес-логики.

## 7.3 Global errors

1. `@RestControllerAdvice` vs `@ControllerAdvice`?

Ответ: `@ControllerAdvice` — базовая аннотация для глобальной обработки исключений во всех контроллерах. Методы, помеченные `@ExceptionHandler`, по умолчанию возвращают имя view-шаблона (как обычный `@Controller`). `@RestControllerAdvice` — это `@ControllerAdvice` + `@ResponseBody`: возвращаемые объекты автоматически сериализуются в JSON (или другой формат по `Accept`-заголовку), без необходимости вручную добавлять `@ResponseBody` к каждому методу. В проекте используется `@RestControllerAdvice`, что соответствует REST API.

2. Как выбирается exception handler?

Ответ: Spring выбирает наиболее специфичный обработчик, тип исключения которого наиболее близок к выброшенному в иерархии классов. Например, если выброшен `EntityNotFoundException`, Spring выберет `@ExceptionHandler(EntityNotFoundException.class)`, а не `@ExceptionHandler(Exception.class)`, хотя оба технически применимы. Если точного совпадения нет, Spring поднимается вверх по иерархии (`RuntimeException` → `Exception`), пока не найдёт подходящий обработчик.

3. Почему обработчик `Exception.class` логически последний?

Ответ: `Exception` — корневой тип всех checked и большинства unchecked исключений. Если поставить его первым или единственным, он перехватит абсолютно все исключения, включая те, для которых есть более специфичные обработчики. В результате `EntityNotFoundException` будет возвращать `500 Internal Server Error` вместо `404 Not Found`, а ошибки валидации — вместо `400 Bad Request`. Обработчик `Exception.class` — это «резервный вариант» для непредвиденных ошибок, поэтому логически он должен идти последним.

## 7.4 Transactions

1. Где ставить `@Transactional` и почему?

Ответ: `@Transactional` нужно ставить на методы сервисного слоя (`@Service`), а не контроллера или репозитория. Сервис — это граница бизнес-операции: одна транзакция должна охватывать всю логику метода (несколько вызовов репозитория, проверки). В проекте все методы `StudentServiceImpl` аннотированы `@Transactional` — `save`, `deleteById` — с режимом по умолчанию (read-write), а `findAll`, `findById` — с `readOnly = true`. Spring создаёт транзакционный прокси-бин и управляет транзакцией через AOP.

2. Что значит `readOnly = true`?

Ответ: `@Transactional(readOnly = true)` сигнализирует провайдеру JPA (Hibernate), что в рамках данной транзакции не будет модификаций данных. Hibernate отключает dirty checking (проверку изменений managed-сущностей) и не генерирует flush-запросы в конце транзакции — это снижает накладные расходы. Некоторые JDBC-драйверы и балансировщики нагрузки могут маршрутизировать `readOnly`-транзакции на реплику для чтения. В `StudentServiceImpl` все методы `findAll`, `findById`, `searchByName`, `searchBySurname`, `countByName` помечены `readOnly = true`.

3. Назовите 3 ловушки transactional-proxy.

Ответ:
- **Self-invocation (самовызов)**: если метод класса вызывает другой `@Transactional`-метод того же класса через `this.method()` (как в `TxPitfallService.outerWithoutAnnotation`), вызов идёт напрямую, минуя Spring-прокси — транзакция не создаётся. Решение: вынести вызываемый метод в отдельный бин или использовать `ApplicationContext.getBean()`.
- **`private`-методы**: Spring-прокси не может перехватить вызов `private`-метода. `@Transactional` на `private`-методе будет проигнорирована — транзакция не откроется.
- **Откат только для `RuntimeException` по умолчанию**: транзакция откатывается автоматически только при `RuntimeException` и `Error`. Если выбросить checked-исключение (например, `IOException`), транзакция будет зафиксирована, если явно не указать `rollbackFor = IOException.class`.

4. При каких исключениях транзакция откатывается по умолчанию?

Ответ: По умолчанию Spring откатывает транзакцию при любом `RuntimeException` (unchecked) и `Error`. Это поведение унаследовано от EJB-конвенции. Checked-исключения (наследники `Exception`, но не `RuntimeException`) по умолчанию **не** вызывают откат — транзакция фиксируется. В `StudentServiceImpl` метод `saveTwoOneBroken` бросает `IllegalStateException` (unchecked) — поэтому если `bad`-студент невалиден, первый `repository.save(good)` будет откатан вместе со вторым.
