# Часть 5: Spring Security

## Задание 5.1

1. Войдите как `user / password`. Можно ли выполнить `POST /api/students`? Какой код?

Ответ: Нет, нельзя. Пользователь `user` имеет только роль `USER`, а `POST /api/students` требует роль `ADMIN` — как в правилах `SecurityConfig` (`.requestMatchers(HttpMethod.POST, "/api/students").hasRole("ADMIN")`), так и в аннотации `@PreAuthorize("hasRole('ADMIN')")` на методе контроллера. В ответ вернётся **403 Forbidden**.

2. Войдите как `admin / password`. Получается ли выполнить `POST /api/students`?

Ответ: Да, успешно. Пользователь `admin` имеет роль `ADMIN`, которая разрешена как в `SecurityConfig`, так и в `@PreAuthorize` на методе `create`. В ответ вернётся **201 Created** с телом созданного студента.

3. Что происходит при `GET /api/students` без аутентификации?

Ответ: Запрос без аутентификации будет отклонён. `GET /api/students` требует роль `USER` или `ADMIN` согласно правилу `.hasAnyRole("USER", "ADMIN")` в `SecurityConfig`. Для REST-запросов (Basic Auth) Spring Security вернёт **401 Unauthorized**; при обращении через браузер — перенаправит на страницу логина.

## Задание 5.2

1. Различие между правилами в `SecurityConfig` и `@PreAuthorize`?

Ответ: Правила в `SecurityConfig` (`.authorizeHttpRequests`) работают на уровне HTTP-запроса — они проверяются фильтром безопасности до того, как запрос вообще достигает контроллера, и оперируют URL-паттернами и HTTP-методами. `@PreAuthorize` — это метод-уровневая безопасность (Method Security), которая работает уже внутри Spring-контекста после диспетчеризации: она проверяет права непосредственно перед вызовом метода бина и позволяет использовать SpEL-выражения (`hasRole`, `hasAnyRole`, `#id == authentication.name` и др.). `SecurityConfig` — грубое сито на входе, `@PreAuthorize` — тонкая настройка на уровне бизнес-логики.

2. Что будет при конфликтующих правилах?

Ответ: Оба уровня проверяются последовательно, и отказ на любом из них блокирует доступ. Если `SecurityConfig` разрешает запрос (например, `permitAll()`), но `@PreAuthorize` на методе требует роль `ADMIN` — пользователь без этой роли получит **403 Forbidden**. Обратное тоже верно: если `SecurityConfig` запрещает URL для роли `USER`, запрос не дойдёт до метода, даже если `@PreAuthorize` его допускал бы. Таким образом, правила не конкурируют — они накладываются, и для успешного доступа нужно пройти оба уровня.
